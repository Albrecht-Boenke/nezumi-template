Warum Design Tokens wichtig sind

Ein Enterprise-UI skaliert nur, wenn visuelle Entscheidungen nicht in jeder Komponente neu getroffen werden. Tokens verhindern Style Drift, reduzieren Refactor-Risiko und machen Dark Mode zu einer semantischen Umschaltung statt zu einer Suche durch TSX-Dateien.

Single Source of TruthDESIGN.md definiert die fachlichen Werte, CSS registriert sie technisch.
Utility-API@theme erzeugt Klassen wie bg-surface, text-text und border-border.
Runtime Theming.dark überschreibt Semantik, ohne Komponenten zu ändern.

Tailwind v4 ist CSS-first

In Tailwind v4 ist @theme die Brücke zwischen CSS Custom Properties und Tailwind Utilities. Für Nezumi ist das wichtig, weil Tokens nicht parallel in TypeScript oder JavaScript gepflegt werden sollen. 

@import "tailwindcss";

@theme {
  --color-brand: var(--color-nezumi-sabi);
  --color-text: var(--color-nezumi-koi);
  --color-surface: var(--color-nezumi-bg);
  --spacing-16: 1rem;
  --radius-lg: 0.5rem;
  --duration-normal: 200ms;
}

Daraus wird eine stabile Markup-API: bg-surface text-text p-16 rounded-lg.

Motion ist ebenfalls ein Token-System

Motion Drift wirkt subtil, aber teuer: Hover, Dropdowns, Dialoge und Page Transitions fuehlen sich uneinheitlich an. Deshalb sind Dauer, Easing und Stagger in Nezumi als @theme-Tokens definiert. 

@theme {
  --duration-fast: 100ms;
  --duration-normal: 200ms;
  --duration-slow: 300ms;
  --duration-slower: 500ms;

  --ease-out: cubic-bezier(0, 0, 0.2, 1);
  --ease-in-out: cubic-bezier(0.4, 0, 0.2, 1);
  --ease-spring: cubic-bezier(0.175, 0.885, 0.32, 1.275);
}

Theming ohne Komponenten-Migration

Dark Mode ersetzt semantische Tokens, nicht Komponentenklassen. Ein Element mit bg-surface text-text border-border bleibt stabil, waehrend .dark die zugrunde liegenden Werte austauscht.

success-bg + success warning-bg + text error-bg + error

Hands-on fuer dieses Projekt

1. Zweck statt Farbe waehlen

Frage zuerst nach der Rolle: Surface, Text, Border, Brand, Status oder Component Interaction. Erst danach wird eine Klasse gewaehlt. 

<Paper className="bg-surface-raised text-text border border-border p-24">
  <Typography variant="title-medium">Statusuebersicht</Typography>
  <Typography variant="body-medium" className="text-text-muted">
    Aktuelle Service-Kennzahlen und offene Pruefpunkte.
  </Typography>
</Paper>

2. Neue Tokens nur bei wiederkehrendem Bedarf

Ein neuer Token ist keine Abkuerzung fuer einen Einzelwert. Er ist ein neuer Begriff im Designsystem und braucht mehrere erwartbare Use Cases.

App-Code konsumiert Tokens. Foundation-Code definiert Tokens. Diese Richtung sollte nicht umgedreht werden.

Data-Flow-Modell

Der Datenfluss ist einseitig. Fachliche Designentscheidungen wandern von DESIGN.md in CSS-Tokens, dann in die gemeinsame UI-Schicht und erst danach in Apps.

1 · DESIGN.md
Visuelle SSOT: Palette, Typografie, Spacing, Radius, Komponentenregeln.
2 · tokens/*
Primitive @theme Tokens: --color-nezumi-*, --spacing-*, --radius-*, --duration-*.
3 · semantic/*
Zweckgebundene Aliase: brand, surface, text, border, success, warning, error.
4 · components/*
Variantenlogik fuer Button, spaeter Card, Input, Dialog oder weitere primitives.
5 · apps/*
AppShell, PageLayout, Section, Paper, Card, Button und Typography konsumieren Utilities.
